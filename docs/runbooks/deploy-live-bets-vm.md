# Runbook — live-bets co-located on the XPredict OCI VM

Deploy the **live-bets** service onto the existing Oracle Cloud VM so
`https://app.<domain>/live` works end-to-end with **zero cold start** — no Render,
no ~40s free-tier delay. Two phases: **F1** (betting-UI only, priority) then **F2**
(HLS video on top).

> Files this runbook touches: `~/xpredict/docker-compose.prod.yml`,
> `~/xpredict/.env.prod`, `~/live-bets/` (separate checkout).
> Ops alias used throughout: `C="docker compose --env-file .env.prod -f docker-compose.prod.yml"`
> (set it per-session — the shell alias does not persist across SSH sessions).

---

## 0. Prerequisites (Pol's OCI console steps — do these FIRST)

1. **Reserve a static public IP** on the OCI instance (Networking → IP Management →
   Reserved Public IPs → Reserve → attach to the instance). Note the new IP — call
   it `<VM_IP>`. The current ephemeral IP (`82.70.92.50`) will be released; DNS will
   break briefly until step 2 propagates.
2. **Namecheap DNS** — update three A-records to `<VM_IP>`:

   | Type | Host | Value   |
   |------|------|---------|
   | A    | `app` | `<VM_IP>` |
   | A    | `api` | `<VM_IP>` |
   | A    | `live` | `<VM_IP>` ← new |

3. **OCI security list** — confirm inbound rules for TCP 80 and 443 exist (they were
   already open for `app`/`api`; no new rules needed for `live` since it's the same
   IP and port).
4. Wait for DNS to propagate (`nslookup live.<domain>` resolves to `<VM_IP>`) before
   starting F1 step 8, otherwise Caddy can't issue the TLS cert.

## 1. Phase 1 — betting-UI live (no video)

**Deliverable:** a logged-in demo player on `https://app.<domain>/live` sees the
live-bets widget, bets on an open round, and the XPredict wallet debits/credits.
No HLS video; the livebets command runs `--no-hls`.

> **Honesty note:** the server-side money path (wallet debit/credit) is proven.
> The **in-browser widget walk** (widget DOM events → XPredict balance island) was
> never run in prod — F1 is the first real end-to-end browser test. Budget for
> first-run surprises in the widget↔backend event wiring.

### 1.1 — SSH in and pull the XPredict repo

```bash
ssh -i ~/.ssh/xpredict_demo_ed25519 ubuntu@<VM_IP>
cd ~/xpredict
git pull   # gets the new compose/Caddyfile/.env.prod.example with livebets services
```

Set the alias for this session:

```bash
C="docker compose --env-file .env.prod -f docker-compose.prod.yml"
```

### 1.2 — Clone (or refresh) the live-bets repo

```bash
# First time:
git clone https://github.com/polito101/live-bets ~/live-bets

# If already cloned:
git -C ~/live-bets pull
```

Confirm the CORS cherry-pick is present on `master`:

```bash
git -C ~/live-bets log --oneline -5
# Should show the CORS commit near the top.

# Belt-and-suspenders: verify the env-gated middleware is in the source:
grep -n "LIVE_BETS_CORS_ORIGINS" ~/live-bets/live_bets/api/app.py
# -> should print at least one match (the env read + CORSMiddleware registration).
```

If the CORS commit is absent, the branch was not yet pushed. Check the live-bets
repo locally; cherry-pick `c379ef8` and push to `origin/master` before continuing.

### 1.3 — Fill `.env.prod` with the new live-bets vars

```bash
# View the documented vars (do not edit this file directly):
grep -A5 "LIVEBETS" ~/xpredict/.env.prod.example

# Open .env.prod (it's gitignored; create from example if brand-new VM):
nano ~/xpredict/.env.prod
```

Add/set the following block. Generate secrets **on the VM**:

```bash
# Generate secrets (run each; copy output into .env.prod):
openssl rand -hex 24   # -> LIVEBETS_DB_PASSWORD  (24 bytes = 48 hex chars)
openssl rand -hex 32   # -> LIVE_BETS_SERVER_KEY  (32 bytes = 64 hex chars, >=32 required)
```

For `LIVE_BETS_ADMIN_PASSWORD` (argon2id hash of a password you choose):

```bash
# Install argon2-cffi if not present, then hash:
pip3 install argon2-cffi --quiet
python3 -c "
from argon2 import PasswordHasher
ph = PasswordHasher(time_cost=2, memory_cost=65536, parallelism=2)
print(ph.hash('YOUR_ADMIN_PASSWORD_HERE'))
"
# Paste the full $argon2id$... string as LIVE_BETS_ADMIN_PASSWORD in .env.prod.
```

`.env.prod` additions:

```dotenv
# live-bets co-located service (F1)
LIVEBETS_DOMAIN=live.<your-domain.com>
NEXT_PUBLIC_LIVEBETS_WIDGET_SRC=https://live.<your-domain.com>/static/widget.js
LIVEBETS_DB_PASSWORD=<output of openssl rand -hex 24>
LIVE_BETS_SERVER_KEY=<output of openssl rand -hex 32>
LIVE_BETS_SERVER_KEY_ID=prod-v1
LIVE_BETS_ADMIN_PASSWORD=<full argon2id hash string>
LIVEBETS_ENABLE_WEBHOOK=false
# Leave blank for now — filled after seeding (step 1.6):
LIVEBETS_API_KEY=
LIVEBETS_DEFAULT_TABLE_ID=
```

`LIVE_BETS_CORS_ORIGINS` is NOT set directly in `.env.prod` — the compose file
passes `${FRONTEND_BASE_URL}` (which you already have set as
`https://app.<your-domain.com>`) to `LIVE_BETS_CORS_ORIGINS` on the livebets
service. No extra line needed.

### 1.4 — Build images

`NEXT_PUBLIC_LIVEBETS_WIDGET_SRC` is baked into the frontend bundle at build time,
so both images must be rebuilt:

```bash
cd ~/xpredict
$C build livebets frontend
# livebets: builds from ~/live-bets (LIVEBETS_BUILD_CONTEXT=../live-bets by default).
# frontend: bakes the new NEXT_PUBLIC_LIVEBETS_WIDGET_SRC into the JS bundle.
# First build: ~5-10 min each. Subsequent builds are layer-cached.
```

### 1.5 — Start live-bets infra and app; verify health

```bash
$C up -d livebets-db livebets-redis livebets
# livebets-db and livebets-redis come up first (depends_on: service_healthy).
# livebets runs `migrate && serve-all --no-hls` on boot — migrations apply automatically.

# Wait ~15s then check:
$C ps livebets-db livebets-redis livebets

# Tail logs (look for "Migrations applied." + "supervisor boot: spawning initial actors"):
$C logs --tail=80 livebets

# Internal health (no host port — exec inside the container):
$C exec livebets sh -c "curl -s localhost:8000/health && echo && curl -s localhost:8000/ready"
# Expected:
#   {"status":"ok"}
#   {"ready":true,"checks":{"postgres":"ok","redis":"ok","hls":"ok","tables":"ok"}}
```

If `ready` shows `postgres: error`, livebets-db is not healthy yet — wait and retry.
If `livebets` restarts in a loop, check `$C logs livebets` for a missing/blank
`LIVE_BETS_SERVER_KEY` or `LIVE_BETS_ADMIN_PASSWORD` — `LIVE_BETS_ENV=prod` fails
fast on placeholder secrets.

### 1.6 — Seed operator + ACTIVE table; mint full-scope key

These commands run **inside the livebets container** against its own Postgres.
`seed_demo_operator.py` is not bundled in the image, so copy it in first.

```bash
# Copy the seed script from the live-bets repo into the running container:
docker cp ~/live-bets/scripts/seed_demo_operator.py \
  $(docker compose --env-file ~/xpredict/.env.prod -f ~/xpredict/docker-compose.prod.yml ps -q livebets):/tmp/seed_demo_operator.py

# Run it against the livebets-db (on the internal compose network):
$C exec \
  -e DATABASE_URL="postgresql://live_bets:${LIVEBETS_DB_PASSWORD}@livebets-db:5432/live_bets" \
  -e BCRYPT_COST=4 \
  livebets python /tmp/seed_demo_operator.py
# Prints: operator_id, a sandbox key (bets:place ONLY — do NOT use this in .env.prod),
# and: "ACTIVE demo table created: <table_id>"
# CAPTURE the <table_id> UUID.
```

> The sandbox key printed by `seed_demo_operator.py` has `bets:place` scope only and
> will 403 on read routes. You need the full-scope key from `bootstrap-admin-key` below.

```bash
# Mint a full-scope operator key (all scopes, unlimited rate — shown ONCE):
$C exec livebets live-bets bootstrap-admin-key \
  --operator-slug xpredict-demo \
  --display-name "XPredict Demo (prod)"
# -> "API key:  lbk_live_<48hex>"
# CAPTURE this key — it is not recoverable after this point.
```

Verify rounds are opening (the make-or-break check — allow ~5s for the Supervisor
to spawn the TableActor):

```bash
$C exec livebets live-bets list-tables
# -> <table_id>  ACTIVE  demo  'Demo Table ...'  betting=30s live=20s settling=10s

$C exec livebets sh -c "curl -s localhost:8000/status"
# -> {"clip_library_size":9,"active_tables":1,...}

$C logs livebets 2>&1 | grep "table actor spawned"
# -> {"event":"table actor spawned","table_id":"<table_id>",...}
# If this line is absent and active_tables=0, the orchestrator died at boot — see Troubleshooting.
```

### 1.7 — Write the key and table_id back into `.env.prod`

```bash
nano ~/xpredict/.env.prod
# Set:
#   LIVEBETS_API_KEY=lbk_live_<48hex>
#   LIVEBETS_DEFAULT_TABLE_ID=<table_id>
```

### 1.8 — Bring up the rest of the stack (backend + frontend + caddy)

```bash
$C up -d backend frontend caddy
# backend is recreated with the new LIVEBETS_API_KEY + LIVEBETS_DEFAULT_TABLE_ID.
# frontend is recreated (already rebuilt in step 1.4).
# caddy picks up LIVEBETS_DOMAIN and provisions the TLS cert for live.<domain>.
# TLS issuance requires DNS to already resolve live.<domain> → <VM_IP> (step 0.4).

# Watch caddy provision the cert (usually <30s once DNS resolves):
$C logs -f caddy
# Look for: "certificate obtained successfully" for live.<domain>
```

### 1.9 — Verify F1 end-to-end

**TLS health check:**

```bash
curl -s https://<live-domain>/health    # {"status":"ok"}  — confirms TLS + Caddy proxy
curl -s https://<live-domain>/ready     # {"ready":true,...}
```

**CORS spot-check** (simulates the browser widget's cross-origin call):

```bash
curl -si "https://<live-domain>/health" \
  -H "Origin: https://app.<your-domain.com>" \
  | grep -i "access-control-allow-origin"
# -> access-control-allow-origin: https://app.<your-domain.com>
```

**Browser walk (Pol's manual step):**

1. Open `https://app.<domain>`, log in as a demo player.
2. Go to `/live` — the widget must render (NOT the "No live table configured" empty
   state). If you see the empty state, jump to Troubleshooting below.
3. Wait for a round to enter **BETTING** (30s window per the demo table config).
4. Place a bet in the widget.
5. Confirm the XPredict wallet balance **drops by the stake** immediately on placement.
6. When the round settles, confirm the wallet credits back (win) or clears (loss).
7. Cross-check the ledger: `$C exec backend psql` or Django shell — look for a
   `livebets_placed` transfer and a `livebets_bets` mirror row with `status=PENDING`
   transitioning to `SETTLED`.

### Troubleshooting (F1)

**`/live` shows "No live table configured" (empty state):**
- `LIVEBETS_DEFAULT_TABLE_ID` is blank or `LIVEBETS_API_KEY` is blank in `.env.prod`.
  Fix the values, then `$C up -d backend` to recreate with the new env (frontend does
  NOT need a rebuild for this — it's a runtime backend env var).
- If the env looks right, confirm the backend was actually recreated (not just
  restarted): `$C ps backend` should show a recent start time after the `up -d`.

**CORS errors in the browser console (blocked cross-origin fetch or WS):**
- `LIVE_BETS_CORS_ORIGINS` on the livebets service is derived from `FRONTEND_BASE_URL`.
  Confirm `FRONTEND_BASE_URL=https://app.<domain>` (exact, no trailing slash) in
  `.env.prod`, then `$C up -d livebets` to recreate. Verify with the curl CORS
  spot-check above. Reference DEMO-RUNBOOK §B5/§B6 for the local analogues.
- If the widget's **WebSocket** connection is rejected cross-origin (browser console
  shows WSS close 1008 or a CORS error on the upgrade), live-bets WS origin handling
  may need a fix on the live-bets side — this was flagged as unconfirmed in the design
  doc. Check `$C logs livebets` for a rejected origin; if so, file against live-bets.

**Orchestrator never opens rounds (`active_tables: 0`, no "table actor spawned"):**
- Schema drift: run `$C exec livebets sh -c "curl -s localhost:8000/ready"` — if
  `postgres: error`, the livebets-db is unavailable. Otherwise check `$C logs livebets`
  for `UndefinedColumnError: live_started_at_pdt`. On a fresh volume this should not
  occur (migrations run cleanly). If it does, `$C rm -sf livebets livebets-db` +
  `docker volume rm xpredict_livebets_pgdata` + restart from step 1.5 (data is not
  yet valuable at this point).

## 2. Phase 2 — HLS video

**Deliverable:** the widget plays live traffic video synced to rounds, over TLS from
`https://<live-domain>`, CORS-clean. Run F1 fully verified before starting F2.

> Do NOT trust `/ready` `hls:ok` — that probe is a no-op and reports ok regardless.
> Verify video by real playback and segment files on disk.

### 2.1 — Confirm ffmpeg version is not 5.1.4

```bash
$C exec livebets ffmpeg -version | head -1
# live-bets rejects ffmpeg 5.1.4 exactly (a known HLS remux bug).
# Any other version is fine. If the output shows 5.1.4, pin a different version
# in the live-bets Dockerfile before building (F2 is gated on this).
```

### 2.2 — Enable HLS in `docker-compose.prod.yml`

On the VM, uncomment three blocks in `docker-compose.prod.yml`:

```bash
nano ~/xpredict/docker-compose.prod.yml
```

In the `livebets:` service, uncomment:

```yaml
    command: ["sh","-c","python -m live_bets.cli migrate && python -m live_bets.cli serve-all --host 0.0.0.0 --port 8000 --bus in-memory --hls --hls-out /app/var/hls"]
    volumes:
      - livebets_clips:/var/lib/live-bets/clips
      - livebets_hls:/app/var/hls
```

In the top-level `volumes:` block, uncomment:

```yaml
  livebets_clips:
  livebets_hls:
```

### 2.3 — Provide real H.264, no-audio clips

Clips MUST be H.264 (no audio). Their **absolute paths on the `livebets_clips` volume
are baked at seed time — never move them after seeding.**

Start with the clips already in the live-bets repo:

```bash
# Identify the livebets_clips volume mountpoint on the host:
docker volume inspect xpredict_livebets_clips --format '{{.Mountpoint}}'
# -> /var/lib/docker/volumes/xpredict_livebets_clips/_data

CLIPS_DIR=$(docker volume inspect xpredict_livebets_clips --format '{{.Mountpoint}}')

# Copy the repo's reference clips:
sudo cp ~/live-bets/var/clip.mp4    $CLIPS_DIR/clip.mp4
sudo cp ~/live-bets/var/sample.mp4  $CLIPS_DIR/sample.mp4
sudo cp ~/live-bets/var/uat_clip.mp4 $CLIPS_DIR/uat_clip.mp4
```

Verify codec (H.264, no audio stream — if `ffprobe` isn't on the host, run via the
livebets container after step 2.5):

```bash
ffprobe -v error -select_streams v:0 -show_entries stream=codec_name,codec_type \
  $CLIPS_DIR/clip.mp4
# codec_name=h264  <-- required; aac/mp3 streams here would mean audio is present
```

If a clip has audio, re-encode it: `ffmpeg -i input.mp4 -c:v copy -an clip.mp4`.

### 2.4 — Seed real clips + bucket membership (detector-free)

The prod image has **no CV detector**, and the synthetic clips from
`seed_demo_operator.py` use `path=/dev/null` which ffmpeg cannot read. You must
insert `clips` rows pointing at the real files and assign them to the ACTIVE table's
buckets.

> **Needs confirmation against the live-bets schema during execution.** The exact
> column names for `clips` and `bucket_membership` should be verified against the
> current migration files (`~/live-bets/migrations/`) before running. The pattern
> below is correct per the design doc; confirm the table DDL matches.

```bash
# Connect to livebets-db:
$C exec livebets-db psql -U live_bets -d live_bets

-- Inside psql:
-- 1. Insert real clips (adjust paths to match where you copied them in step 2.3):
INSERT INTO clips (path, duration_s, vehicle_count) VALUES
  ('/var/lib/live-bets/clips/clip.mp4',     30, 5),
  ('/var/lib/live-bets/clips/sample.mp4',   30, 8),
  ('/var/lib/live-bets/clips/uat_clip.mp4', 30, 3)
ON CONFLICT (path) DO NOTHING
RETURNING id, path;
-- Note the returned clip ids.

-- 2. Find the buckets for the ACTIVE demo table:
SELECT id, label FROM buckets
  WHERE table_id = '<table_id>';  -- from step 1.6

-- 3. Assign the clips to each bucket (one row per clip-bucket pair):
INSERT INTO bucket_membership (bucket_id, clip_id)
VALUES
  ('<bucket_id>', '<clip_id_1>'),
  ('<bucket_id>', '<clip_id_2>'),
  ('<bucket_id>', '<clip_id_3>')
ON CONFLICT DO NOTHING;
-- Repeat for each bucket that should have clips.
\q
```

> If `clips.duration_s` or `clips.vehicle_count` are not the real column names,
> check `~/live-bets/migrations/` for the `clips` table DDL and adjust accordingly.

### 2.5 — Apply the command + volumes; verify HLS output

```bash
$C up -d livebets
# Recreates the livebets container with the new command (--hls --hls-out) + volumes.

# Watch for ffmpeg activity during a LIVE round:
$C logs -f livebets
# Look for: "ffmpeg" process lines, "segment" writes, or "channel start" events.

# After a LIVE round completes (~20s after BETTING ends), check for segment files:
$C exec livebets sh -c "ls -R /app/var/hls"
# Expected: /app/var/hls/<table_id>/master.m3u8 + seg-*.m4s files
```

### 2.6 — Verify HLS over TLS

```bash
# 200 + CORS header (HLS routes emit Access-Control-Allow-Origin: * unconditionally):
curl -sI "https://<live-domain>/stream/<table_id>/master.m3u8" \
  | grep -E "HTTP|access-control"
# -> HTTP/2 200
# -> access-control-allow-origin: *
```

**Browser verification (Pol's manual step):** open `/live` during a LIVE round and
confirm the widget plays moving video synced to the round. No segment files on disk
= ffmpeg not running = check the clip paths and bucket assignment in step 2.4.

## 3. Ops & rollback

### Routine operations

```bash
C="docker compose --env-file .env.prod -f docker-compose.prod.yml"

$C ps                          # status + health of all services
$C logs -f livebets            # tail live-bets logs
$C logs -f caddy               # TLS cert events, proxy errors
$C restart livebets            # restart without config change (keeps volumes)
$C exec livebets live-bets list-tables   # confirm table state
$C exec livebets live-bets migrate       # re-run migrations (idempotent)
```

### Disable live-bets (revert to graceful empty state)

If you need to take live-bets offline without touching anything else, blank the
table ID in `.env.prod` and recreate only the backend:

```bash
# In .env.prod, set:
#   LIVEBETS_DEFAULT_TABLE_ID=
nano ~/xpredict/.env.prod

$C up -d backend
# The /live page now shows "No live table configured" — no errors, no crash.
# The livebets, livebets-db, livebets-redis containers keep running; re-enable
# by restoring LIVEBETS_DEFAULT_TABLE_ID and re-running `$C up -d backend`.
```

### Full teardown of live-bets only

Stops and removes the three live-bets containers; leaves XPredict running:

```bash
$C rm -sf livebets livebets-db livebets-redis

# To also drop the data volumes (irreversible — operator key + table data gone):
docker volume rm xpredict_livebets_pgdata xpredict_livebets_redis_data
# F2 volumes (if created):
# docker volume rm xpredict_livebets_clips xpredict_livebets_hls

# After teardown, blank both vars in .env.prod and recreate the backend so the
# /live page fails gracefully:
$C up -d backend
```

### Update live-bets code

```bash
git -C ~/live-bets pull          # pull new commits on master
$C build livebets                # rebuild the image
$C up -d livebets                # recreate (migrations re-run on boot)
```
