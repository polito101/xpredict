# XPredict v1.0 — Milestone Audit

**Audited:** 2026-06-02 · **Status:** ⚠ `gaps_found` · **Scope:** Phases 1–11 (all merged to `main`)

## Executive summary

The **backend is essentially complete and well-tested** — at the code level **68 of 69 requirements are implemented**, with real FOR-UPDATE locking, single-transaction double-entry, idempotent settlement, compensating-entry reversal, Polymarket sync + auto-resolution reusing the same `SettlementService`, and integration tests (including injected mid-transaction-failure rollback tests) backing the financial core.

But the milestone's own definition of done is an **operator-ready demo**, and the audit found **3 blockers that break every core demoable flow at the UI seam**. This is the classic existence-vs-integration gap: ~10 requirements are "complete in isolation" (endpoints exist, admin-gated, tested) yet unreachable because no frontend wires them into a user-facing flow. Historically this traces to Phase 5 (built ad-hoc by Agustin via PR #8) whose `.continue-here.md` explicitly noted "resolution display / admin two-step resolve UI NOT built", plus the Phase-4 admin market UI that fell into no-man's-land (admin-nav labels it "Phase 10 / deferred", but Phase 10 shipped branding + KPI instead).

**Verdict:** Do not archive/tag v1.0 yet. One focused closure phase (admin market-ops UI + player resolution display + persist the winner) makes the demo genuinely operator-ready.

## The 3 blockers

### BLOCKER-1 — Player resolution display (STL-06) is genuinely missing
The winning outcome, resolution source ("Polymarket UMA" / "Operator: {name}"), public justification, and the player's own payout are surfaced on **no player page**. Root cause: `settlement/adapters.py::mark_resolved` discards `winning_outcome_id` — the winner is persisted **only** in the admin-gated audit log; `markets.winning_outcome_id` / `resolution_source` / `resolution_justification` columns don't exist. Compounding: `markets/router.py::get_market_public` (line ~164) **404s any RESOLVED market**, so the player literally sees "Market not found". Fixing the UI alone is insufficient — it needs a small backend data change (migration + persist on resolve).
**Affected:** STL-06. Breaks the player-visible end of Flows 1 & 2.

### BLOCKER-2 — No admin UI to resolve / reverse / force-settle
`/admin/markets/{id}/resolve|reverse|force-settle` all exist, are admin-gated, and are tested — but **no frontend calls them**. An operator cannot resolve a house market or reverse/override from the product.
**Affected:** STL-02, STL-07, ADM-05, ADM-06. Breaks Flow 1 (admin-resolve step) and Flow 3.

### BLOCKER-3 — No admin market-management UI at all
Full house-market lifecycle backend (create/list/edit/close, 423 criteria-lock) exists and is tested; **zero frontend**. `admin-nav.tsx` hardcodes "Markets" as a disabled placeholder. Without a create-market UI, the only way to get a house market into the demo is raw API.
**Affected:** ADM-01, ADM-02, ADM-03, ADM-04, ADM-07. Breaks Flow 3 (and the house side of Flow 1).
