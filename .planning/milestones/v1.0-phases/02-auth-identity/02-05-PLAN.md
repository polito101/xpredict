      ```typescript
      import { NextRequest, NextResponse } from 'next/server'
      import { jwtVerify } from 'jose'

      const ADMIN_PROTECTED = /^\/admin(\/|$)/
      const ADMIN_LOGIN = '/admin/login'

      export async function middleware(req: NextRequest) {
        if (!ADMIN_PROTECTED.test(req.nextUrl.pathname)) return NextResponse.next()
        if (req.nextUrl.pathname === ADMIN_LOGIN) return NextResponse.next()

        const token = req.cookies.get('admin_jwt')?.value
        if (!token) return NextResponse.redirect(new URL(ADMIN_LOGIN, req.url))

        try {
          const secret = new TextEncoder().encode(process.env.ADMIN_JWT_PUBLIC_SECRET!)
          await jwtVerify(token, secret, { algorithms: ['HS256'] })
          return NextResponse.next()
        } catch {
          return NextResponse.redirect(new URL(ADMIN_LOGIN, req.url))
        }
      }

      export const config = {
        matcher: ['/admin/:path*'],
      }
      ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Edge runtime middleware → admin_jwt cookie verification | Middleware verifies HS256 signature with `ADMIN_JWT_PUBLIC_SECRET`; if signature/expiry fails, redirect. NO DB access in middleware (Edge runtime constraint + Anti-pattern RESEARCH line 923) |
| Server Action (adminLoginAction) → FastAPI /admin/auth/login | Cross-origin POST to backend; Bearer JSON response parsed and re-stored as HttpOnly cookie scoped to `/admin/` |
| admin_jwt cookie path `/admin/` | Cookie is NOT sent to /auth/* or / routes — browser-side defense-in-depth so admin token cannot leak to player surface via accidental fetch |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-47 | Spoofing | Middleware grants access on a forged admin_jwt (signature not verified) | mitigate | jose.jwtVerify with `algorithms: ['HS256']` (Pitfall — Algorithm confusion: NEVER pass undefined algorithms; jose throws if header alg doesn't match); test_redirects_on_invalid_jwt_signature asserts |
| T-02-48 | Spoofing | Middleware bypassed by direct API call to /admin/* (curl skipping the frontend) | mitigate | FastAPI `current_active_admin` dependency is the AUTHORITATIVE gate; middleware is OPTIMISTIC only (RESEARCH lines 913-914); 02-03 test_player_login_does_not_grant_admin asserts backend enforcement |
| T-02-49 | Tampering | ADMIN_JWT_PUBLIC_SECRET committed to git via .env.example | mitigate | Placeholder value `change-me-32+chars-must-match-backend-SECRET_KEY` is gitleaks-safe (verified via Phase 1 allowlist); pre-commit gate runs on every commit |
| T-02-50 | Information Disclosure | admin_jwt cookie sent on /auth/* requests (token leak via player API) | mitigate | Cookie scoped with `path: '/admin'`; browser does NOT include it on / or /auth/* requests; test that adminLoginAction passes the path option is covered by the Server Action unit test |
| T-02-51 | Tampering | XSS injects script that reads admin_jwt | mitigate | admin_jwt is HttpOnly (Server Action sets `httpOnly: true`); JS cannot read it; React escapes user input by default |
| T-02-52 | Information Disclosure | Middleware logs the JWT payload via console.log | mitigate | Middleware code does NOT log anything (no console.log calls); structlog scrub runs server-side only — Edge runtime has no logging path |
| T-02-53 | Elevation of Privilege | ADMIN_JWT_PUBLIC_SECRET differs from backend SECRET_KEY → all admin sessions invalid OR middleware accepts unsigned tokens | mitigate | Documentation in .env.example explicitly states "Must equal backend SECRET_KEY"; deployment runbook (Phase 11) will check parity; if mismatch, middleware always redirects (fail-closed) — verified by test_redirects_on_invalid_jwt_signature |
| T-02-54 | Denial of Service | jose.jwtVerify is CPU-heavy and runs on every /admin/* request | accept | HS256 is symmetric and fast (~µs per verify); Edge runtime caches the encoded secret; far cheaper than a DB round-trip (Anti-pattern in RESEARCH line 923 mitigates the DB-in-middleware DoS path) |
| T-02-55 | Information Disclosure | adminLoginAction surfaces the raw access_token to the client (via redirect URL or returned action state) | mitigate | Token is set via cookies().set() server-side; the action returns `{success: true}` or an error object — never the token; the cookie is HttpOnly so the client cannot read it |
| T-02-56 | Repudiation | No client-side audit of admin actions | accept | Frontend has no audit obligation; backend audits every /admin/* state mutation (Phase 4+ admin endpoints will use AuditService.record per CONVENTIONS §3) |
| T-02-SC | Tampering | Supply-chain — no new packages installed in this plan | accept | jose was added in 02-04; this plan reuses it. No new pip/pnpm installs |
</threat_model>
