<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Next.js Server Action | Form data crosses into the SSR runtime; zod schemas validate at this boundary as a UX-only pre-check (backend re-validates) |
| Next.js Server Action → FastAPI backend | Backend URL configured via `BACKEND_URL` env var; cross-origin POST with `credentials: 'include'` so cookies flow back |
| Set-Cookie response header → browser | Server Action re-sets the cookie via `cookies().set(...)` so Next.js can forward it; HttpOnly + SameSite + Secure attributes carried verbatim from the backend response |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-37 | Tampering | Client-side zod schema "skipped" by hostile user → weak password reaches backend | mitigate | Backend `validate_password` is authoritative (RESEARCH §"Architectural Responsibility Map" line 98); client zod is UX-only; test_weak_password_rejected (02-02 Task 2) asserts backend rejection |
| T-02-38 | Information Disclosure | Forgot-password UI reveals whether email exists via different messages | mitigate | forgotPasswordAction returns the same generic success message regardless of response; tested via auth.test.ts |
| T-02-39 | Information Disclosure | BACKEND_URL leaks the FastAPI internal hostname into the client bundle | mitigate | BACKEND_URL is server-only (no `NEXT_PUBLIC_` prefix); Server Actions read it from `process.env` at runtime; client cannot import it |
| T-02-40 | Tampering | Client-side localStorage / sessionStorage holds the session token (xss bait) | mitigate | The cookie is HttpOnly — JS cannot read it. Frontend NEVER persists the session token client-side; auth state is inferred from the cookie's presence (or via a `/auth/users/me` round-trip) |
| T-02-41 | Tampering | XSS injection via display_name or email reflection in the UI | mitigate | React escapes all user input by default; no `dangerouslySetInnerHTML` anywhere in the auth pages; shadcn primitives use safe escape |
| T-02-42 | Spoofing | CSRF: malicious site forces the browser to POST to /auth/login or /auth/register | mitigate (partial) | Cookie is `SameSite=Lax` (Pitfall 3 cross-mitigation); for cookie-authenticated state-changing POSTs from cross-origin contexts, Phase 11 adds CSRF token if needed; v1 demo is same-origin (Next.js + FastAPI behind reverse proxy in staging) — accepted per RESEARCH line 1523 |
| T-02-43 | Denial of Service | Form submission spammed → backend rate limit returns 429 | mitigate | Server Actions surface the 429 to the user as a friendly "Too many attempts" message instead of a generic error; rate limit is enforced server-side (D-14) |
