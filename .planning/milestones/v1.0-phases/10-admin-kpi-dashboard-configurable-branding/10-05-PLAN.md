<objective>
Slice B (frontend) — Runtime theming consumption. The player root layout (async Server Component) awaits the public GET /branding/current per navigation and injects a <style>:root{--brand-primary;--brand-secondary}</style> block from the server-validated hexes; globals.css gains the --brand-* tokens (with safe fallbacks) mapped through @theme inline; the player header consumes the brand name + logo. Implements D-10, the UI-SPEC runtime-theming + accessibility contract, and ADD-06 (the white-label live re-skin). Depends on Plan 10-01's public /branding/current + /branding/logo.

End-to-end (the demo money-shot, SC#5): an operator changes the palette in /admin/branding → a player navigates to any page → the new --brand-* vars are injected → the UI re-skins with NO rebuild/redeploy.

Purpose: this is what makes branding "real" rather than a demo trap — per-navigation fetch, no static color inlining. The security guard is that the hexes are server-validated (^#[0-9a-fA-F]{6}$) in Plan 10-01 BEFORE persist AND the layout treats them as opaque validated tokens (never re-derives/concatenates untrusted strings).
Output: globals.css brand tokens + the async root layout injection + a public branding fetch helper + a BrandLogo header component + a test asserting the fallback + the injected-hex shape.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| public /branding/current → player root layout | Unauthenticated read of operator-chosen branding rendered into every player page. |
| operator-controlled hex → player <style> :root | The single highest-risk injection point: operator palette interpolated into a <style> tag served to all players. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering / Elevation (stored CSS/HTML injection) | hex interpolated into the player <style> :root block | mitigate | The hexes are server-validated `^#[0-9a-fA-F]{6}$` in Plan 10-01 BEFORE persist AND the layout treats them as opaque validated 6-digit tokens; a valid hex cannot contain `<`, `>`, `}`, or quotes so no `</style>` break-out is possible. The layout interpolates ONLY b.primary_hex/b.secondary_hex and never concatenates any other untrusted string into the block. |
| T-10-02 | Elevation (XSS) | logo SVG rendered in the player header | mitigate | BrandLogo renders the logo via `<img src="/branding/logo">` only (SVG-in-img does not execute script); the logo is served with X-Content-Type-Options: nosniff (Plan 10-01) and is NEVER inlined into the DOM as markup. |
| T-10-17 | Denial of Service (UX) | /branding/current unreachable → unbranded/broken player UI | mitigate | DEFAULT_BRANDING fallback + the :root CSS-var defaults in globals.css apply when the fetch fails; both defaults are ≥4.5:1 on white. The player UI is never unbranded-broken (UI-SPEC accessibility guardrail #3). |
| T-10-18 | Tampering (legibility) | a bad operator palette makes player text unreadable | accept | Structural guardrail (UI-SPEC A-PALETTE): brand color is an accent on neutral surfaces, legibility-critical header text/CTAs keep zinc ink, validation guarantees a real 6-digit hex. A runtime WCAG contrast engine is deferred to v2 (documented, Pol-flagged). |
| T-10-SC | Tampering | npm/pip/cargo installs | accept | No new packages this phase (RESEARCH §Standard Stack — all frontend deps already present; zero new installs). No legitimacy checkpoint required. |
</threat_model>
