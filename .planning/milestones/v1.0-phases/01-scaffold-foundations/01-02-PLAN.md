<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → server (Next.js) | unauthenticated HTTP — Phase 1 ships only `/`, `/api/healthz`, `/api/sentry-test`; no auth surface |
| process.env → client bundle | `NEXT_PUBLIC_SENTRY_DSN` is shipped to the browser; treated as a public token (write-only DSN) per D-14 |
| frontend → backend | Phase 1 frontend does NOT call the backend; the link is established in Phase 2 |
| Sentry SDK → Sentry SaaS | events leave both server and client tagged `service=frontend`; PII suppressed by default in `@sentry/nextjs` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | client bundle exposes `NEXT_PUBLIC_*` env vars | accept | Sentry DSN is a write-only public token per Sentry security model; D-14 documents the choice; gitleaks blocks accidental commit of non-`NEXT_PUBLIC_*` secrets via `.gitleaks.toml` (PLT-04) |
| T-02-02 | Information Disclosure | `/api/sentry-test` returns 500 with stack trace in dev | accept | Phase 1 ships endpoint naked per D-29; cost is ~3 noise events during testing; the 5k/month Sentry free tier absorbs it. Phase 11 gates it behind `?key=` if needed. |
| T-02-03 | Tampering | Sentry events from frontend untagged → can't be filtered from backend events | mitigate | `initialScope.tags.service = "frontend"` set in BOTH `instrumentation.ts` AND `instrumentation-client.ts` (Pattern 5c); test asserts the tag is present |
| T-02-04 | Denial of Service | `/api/healthz` doing heavy work fails docker-compose healthcheck under load | mitigate | endpoint is statically `Response.json({status: "ok"})` — no DB, no fetch, no async I/O; runs in <1ms |

</threat_model>
