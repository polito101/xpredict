<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner -> generated `.env` | Staging-style secrets are written into the runner filesystem; must never be logged or committed |
| application source -> production boot | A hardcoded dev URL / `ENVIRONMENT=dev` that survives to a prod deploy is a config-tampering vector |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01-01 | Information Disclosure | dry-run `.env` heredoc step | mitigate | Generate `.env` in-job; never `cat`/`echo` its contents to the log; `.env` is gitignored so it cannot be committed; values are throwaway placeholders, not real secrets |
| T-11-01-02 | Tampering/Config | hardcoded dev URL leaking to prod | mitigate | `bin/check_no_dev_config.sh` fails the build on `localhost`/`127.0.0.1`/`ENVIRONMENT=dev` in `backend/app` + `frontend/src` (the SC#3 guard) |
| T-11-01-03 | Denial of Service | compose port collision in CI | accept | GH runner host is clean; default compose ports are free; documented in RESEARCH Pitfall 4. Override-drop-ports fallback only if a collision ever appears |
| T-11-01-SC | Tampering | npm/pip installs | accept | This plan adds NO package installs (reuses existing `uv sync --frozen`); the package-legitimacy gate lives in plan 11-02 |
</threat_model>
