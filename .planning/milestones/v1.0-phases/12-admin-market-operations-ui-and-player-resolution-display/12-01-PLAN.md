<objective>
Land the ONE additive backend change for the v1.0 closure phase: persist the winning outcome on resolve (STL-06 root cause) and add the per-market stake-limit columns (BET-06 storage). This is the cross-cutting foundation both player-display and per-market-enforcement slices build on.

Purpose: STL-06 is currently broken at the data layer — `mark_resolved` discards `winning_outcome_id` (winner lives only in the admin-gated audit log) and `get_market_public` 404s any RESOLVED market, so the player literally sees "Market not found". This plan persists the winner inside the existing settlement ACID transaction, exposes the resolution fields publicly, and adds the nullable per-market stake columns — without touching the tested ledger math.

Output: migration `0010`, the 5 new `markets` columns, the extended `MarketRead`/`MarketCreate`/`MarketUpdate`, the relaxed public-read guard, the atomic `mark_resolved` ripple across the Protocol + real adapter + service call site + 6 test fakes, and an applied migration.

Decisions / divergences (surface to Pol at plan review):
- **BET-06 storage diverges from the literal requirement text.** REQUIREMENTS/ROADMAP say per-market limits "via TenantConfig". `TenantConfig` is a verified SINGLE-ROW global table (`UNIQUE(tenant_id)`) — structurally unfit for per-MARKET values. Per RESEARCH A1 + PATTERNS, we store NULLABLE `min_stake`/`max_stake` columns ON `markets` (NULL = fall back to the existing global `BET_MIN_STAKE`/`BET_MAX_STAKE` config). The global config constants are NOT removed; they remain the default. This is the one place the build diverges from the requirement wording, on evidence.
- **`resolution_source` is stored as a stable token** (`"HOUSE"` / `"POLYMARKET_UMA"`), derived in `resolve_market` from `actor_user_id` (None => auto/Polymarket path; set => admin/house). The audit row already records the resolver UUID; the new column is a denormalized, publicly-readable projection. The player-facing "Operator: {name}" display name is resolved in the frontend slice (12-04) — this plan stores the token only.
- **Pre-Phase-12 RESOLVED markets** (RESEARCH A3): for a fresh demo DB this is moot; for an existing DB, historical resolved markets would show a degraded panel (no winner label) because their winner is audit-log-only. We do NOT add a backfill UPDATE in this plan (keeps the migration purely additive + reversible; the demo runs on a fresh/seeded DB). If Pol wants historical markets to render fully, a follow-up backfill task can read `audit_log.payload.winning_outcome` — flagged, not silently attempted.
</objective>
