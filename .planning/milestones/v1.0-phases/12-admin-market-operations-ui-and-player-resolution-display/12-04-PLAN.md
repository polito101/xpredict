Render the player resolution display (STL-06): when a market is RESOLVED, the player-detail page shows the winning outcome, resolution source attribution, public justification, settlement timestamp, and — for a logged-in bettor — their own payout/loss, in place of the order-entry form.

Output: the extended `MarketDetail` type (4 resolution fields), `MarketResolutionPanel`, its test, and the RESOLVED branch + own-result fetch in `markets/[slug]/page.tsx`.

So the panel renders `"POLYMARKET_UMA"` → "Polymarket UMA" (+ the existing `SourceBadge` source link) and `"HOUSE"` → "Operator" (without a name — the public read has no admin join). Write the copy defensively: if a future backend adds a resolver display-name field to `MarketRead`, the panel uses "Operator: {name}"; until then it shows "Operator". This is the documented, non-blocking fallback — flag it in the SUMMARY so Pol can decide whether to add the name snapshot later.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| player cookie -> /bets/me/portfolio | the own-payout read is self-scoped by the player's own HttpOnly cookie; it must never expose another user's result |
| public market read -> rendered justification | the public justification is operator-authored text shown to all players |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-12-11 | Information Disclosure | resolution panel leaking another user's payout | mitigate | The own-result comes from `/bets/me/portfolio` (self-scoped by the player's own `xpredict_session` cookie, no `user_id` param), filtered by `market_id` after the SSR fetch; the cookie value never reaches the client, only the rendered result |
| T-12-12 | Tampering / XSS | HTML injection via the public justification | mitigate | Justification is rendered as escaped React text (`{justification}`); NO `dangerouslySetInnerHTML`; a `<b>`-containing justification renders literal characters (asserted in the panel test) |
| T-12-13 | Information Disclosure | session cookie crossing to client JS | mitigate | The cookie is read server-side in the Server Component and forwarded server-side; only the boolean `isAuthenticated` + the rendered result reach the client (established `portfolio/page.tsx` pattern) |
| T-12-SC | Tampering | npm/pip/cargo installs | mitigate | Zero installs (RESEARCH §Package Legitimacy Audit); no install task present |
</threat_model>
