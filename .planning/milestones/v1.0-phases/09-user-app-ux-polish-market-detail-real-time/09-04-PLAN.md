<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → placeBetAction (Server Action) | The form submits stake/outcome; the action runs server-side |
| placeBetAction → backend `POST /bets` | Cookie-forwarded authenticated bet placement |
| SSR fetch → backend public read endpoints | Server Component reads market/history/activity by slug |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-12 | Elevation of Privilege | order-entry form / placeBetAction | mitigate | The form does NOT bypass any server gate: `place_bet` is gated by `current_betting_player` (verified-email + not-banned + balance + stake-limits) server-side; the client zod is pre-flight UX only. placeBetAction forwards the `xpredict_session` cookie — it cannot place a bet for another user (no user_id param; the backend resolves the player from the session). 401/403 are surfaced as the login / verify / banned inline messages, never silently bypassed (RESEARCH Architectural Responsibility Map). |
| T-09-13 | Spoofing | session cookie handling | mitigate | placeBetAction reads the existing HttpOnly `xpredict_session` via `next/headers cookies()` and forwards it server-side (mirrors `portfolio/page.tsx`); the cookie never enters client JS. `BACKEND_URL` (server-only, no `NEXT_PUBLIC_`) is used for the POST so no backend secret leaks to the bundle. |
| T-09-14 | Information Disclosure | recent-activity feed render | mitigate | The feed renders only the anonymized `ActivityItem` shape from the backend (`outcome`, `amount`, `created_at`) — there is no user-identity field to render. The component never derives or displays a username/id (defense-in-depth on top of the server-side anonymization in Plan 02). |
| T-09-15 | Information Disclosure | bet error copy | accept | Inline error copy is intentionally specific (balance vs closed vs unverified vs banned vs stake-limit) per the UX trust requirement (no generic toast). None of these messages leak another user's data or a secret — they describe the caller's own request state. |
| T-09-SC | Tampering | npm/pip/cargo installs | mitigate | No NEW installs in this plan — the Radix dialog/select packages were added + vetted in Plan 03 (blocking checkpoint there). This plan only composes components + adds a Server Action. No `[ASSUMED]`/`[SUS]` packages → no legitimacy checkpoint. |
</threat_model>
