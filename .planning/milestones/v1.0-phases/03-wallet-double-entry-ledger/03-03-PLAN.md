<objective>
Satisfy SC#1 (WAL-01): a `user_wallet` account is created in the SAME transaction as the user row
on registration. RESEARCH proved (by reading installed fastapi-users 15.0.5 source) that the stock
`SQLAlchemyUserDatabase.create()` calls `session.commit()` BEFORE `on_after_register` fires — so the
existing `on_after_register` hook CANNOT host same-transaction work. The fix is RESEARCH Option A:
override `UserManager.create()` to co-insert the wallet on the adapter's session between the user
INSERT and a single commit.

This is the one genuinely non-obvious wiring decision in the phase. The locked decision (per RESEARCH
A1) is Option A — minimal blast radius, keeps the change inside the already-customized `UserManager` —
and `WalletService.create_wallet` (built in 03-02) does add+flush only, never commits.

Purpose: every new player automatically owns exactly one wallet, atomically — the foundation the
recharge (03-04) and reads (03-05) surfaces assume exists.
Output: a `UserManager.create()` override + the SC#1 integration test.
</objective>
