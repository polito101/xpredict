<objective>
Build the backend half of MKT-03's data surface: a public price-history endpoint (with server-side downsampling beyond 7 days) and a public recent-activity endpoint (anonymized server-side). These feed the Recharts YES-line chart and the anonymized activity feed on the market detail page. The detail payload itself (`MarketRead`) already returns `resolution_criteria` + `outcomes[].current_odds`, so no change is needed there — only the two new read endpoints.

Purpose: A player opening `/markets/{slug}` must get a time-series for the chart and a feed of recent bets without the browser ever receiving raw 30-day snapshots or any user identity. Anonymization is a privacy requirement (CONTEXT Area 1) and MUST happen server-side. This is the MKT-03 data slice of the phase user story (see 09-01 Phase Goal).
Output: `PricePoint`/`PriceHistoryResponse`/`ActivityItem` schemas, `MarketService.price_history(slug, window)` + `recent_activity(slug, 20)`, two public GET endpoints on `public_market_router`, and Wave-0 tests covering the string serialization, 30d downsampling, and the no-user-identity negative assertion.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-05 | Information Disclosure | `GET /{slug}/activity` | mitigate | Anonymization is SERVER-SIDE: the query selects only `b.stake`, `b.created_at`, `o.label` and the `ActivityItem` schema has no user field — `user_id`/email/display_name are never selected, never serialized. A negative test asserts no user-identity key appears in the JSON. (CONTEXT Area 1, RESEARCH Pattern 8.) |
| T-09-06 | Information Disclosure | money/odds serialization | mitigate | `probability` and `amount` serialize as JSON strings via `field_serializer` (SP-1) — no lossy float, consistent with the project-wide money-as-string firewall. No PII in price-history (it is odds only). |
| T-09-07 | Denial of Service | `price-history?window=30d` raw-snapshot flood | mitigate | The 30d window is DOWNSAMPLED to hourly buckets server-side (Pattern 5); the browser never receives ~8640 raw points. `OddsSnapshot.outcome_id`/`snapshot_at` are indexed so the bucketed query is bounded. Read-only — no write amplification. |
| T-09-08 | Tampering | window query param | mitigate | `window` is validated against the allowlist `{24h,7d,30d}` (422 on anything else); no raw interval string reaches the SQL — the cutoff is computed from the validated enum-like value, not interpolated. |
| T-09-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this plan — read-only endpoints over existing models. No `[ASSUMED]`/`[SUS]` packages → no legitimacy checkpoint. |
</threat_model>
