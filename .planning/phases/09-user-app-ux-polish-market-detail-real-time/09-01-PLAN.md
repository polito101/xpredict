<objective>
Build the backend half of MKT-04: real-time WebSocket price broadcasting. Lift the VALIDATED spike 003 pipeline (FastAPI native WebSocket + `redis.asyncio` pub/sub) into `app/realtime/`, wire the subscriber into the existing app lifespan, and add the two producer hooks that publish odds-change deltas — the admin odds edit (producer site #1) and the Polymarket poll (producer site #2).

Purpose: Players watching a market detail page must see odds move within 2s when a Polymarket poll detects movement or an admin edits house odds — with no browser polling (per CONTEXT Area 3). This plan delivers the producer→Redis→subscriber→WS-client pipeline end-to-end so the frontend slice (Plan 03) can connect.
Output: `app/realtime/{manager,subscriber,publisher,router}.py`, an extended `app/main.py` lifespan + router include, publish hooks in `MarketService.update_market` and the poll path, and the Wave-0 integration test scaffolds for the fan-out, isolation, reconnect, and both producer hooks.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → WS `/ws/markets/{id}` | Untrusted, unauthenticated public clients open long-lived sockets |
| Celery/admin process → Redis pub/sub | Internal producers publish odds deltas across the process boundary |
| Redis pub/sub → uvicorn subscriber | The subscriber decodes arbitrary channel payloads and fans out |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-01 | Denial of Service | Public WS endpoint `/ws/markets/{id}` | mitigate | No per-client server state grows unbounded: `ConnectionManager` stores only the live socket set keyed by market_id; `disconnect()` runs in `finally` on every `WebSocketDisconnect` and dead sockets are pruned during `broadcast`. The subscriber is one task per worker (started in lifespan, cancelled on shutdown — Pitfall 4), not per-connection. Connection-count caps + handshake rate-limiting are deferred to Phase 11 hardening (documented), not invented here. |
| T-09-02 | Information Disclosure | WS broadcast payload | mitigate | The production delta is exactly `{type, market_id, outcomes:[{outcome_id, odds}], ts}` — odds are already-public data (same as `GET /api/v1/markets`). NO user identity, NO PII. The spike's dev-only `_latency_ms`/`_server_ts`/forensic event_log are STRIPPED before production (SP-4). |
| T-09-03 | Tampering | Odds delta published before DB commit | mitigate | `publish_odds_change` is called ONLY post-commit at both producer sites (Pitfall 3): the admin edit publishes in the router after `session.commit()`; the poll publishes after its `session.commit()` and only for markets whose per-market sync committed (the loop rolls back + `continue`s on IntegrityError). Clients never render a rolled-back price. |
| T-09-04 | Spoofing | Inbound WS messages from clients | accept | The WS is read-only broadcast; the only inbound message handled is `"ping"` → `pong`. Any other inbound text is ignored. A client cannot inject a price (it can only receive). Bet placement stays on the authenticated REST endpoint (Plan 04), not the socket. |
| T-09-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this plan — `fastapi[standard]`/`uvicorn[standard]`/`redis>=5.0` are all already pinned (RESEARCH Standard Stack, ZERO new backend deps). `websockets` is test-only and bundled by `fastapi[standard]`. No `[ASSUMED]`/`[SUS]` packages → no legitimacy checkpoint required. |
</threat_model>
