# Phase 9 — UI Design Contract

> Visual and interaction contract for the player-facing market detail page (`MKT-03`) and
> real-time WebSocket price updates (`MKT-04`). Generated by gsd-ui-researcher, verified by
> gsd-ui-checker.
>
> **Prime directive:** this phase must look and feel like a seamless continuation of the
> Phase 6 market list. Every token below was extracted from the existing codebase
> (`frontend/src/app/globals.css`, `components/ui/*`, `market-card.tsx`, `odds-display.tsx`,
> `portfolio/page.tsx`). Do NOT introduce new palettes, fonts, or radii. Reuse first; extend
> only where this phase adds genuinely new surfaces (chart, order-entry form, live indicator).

## Spacing Scale

Declared values (multiples of 4 — matches existing `p-6` / `gap-4` / `space-y-2` usage):

| Token | Value | Tailwind | Usage in this phase |
|-------|-------|----------|---------------------|
| xs | 4px | `1` / `gap-1` | Live-dot to label gap, inline `·` separators in activity rows |
| sm | 8px | `2` / `gap-2` / `space-y-2` | Label→value stacking, form field internal spacing (matches `FormItem space-y-2`) |
| md | 16px | `4` / `gap-4` | Grid gap between cards (matches `market-list` `gap-4`), activity-row vertical rhythm |
| lg | 24px | `6` / `p-6` / `gap-6` | Card padding (matches every `Card*` `p-6`), page section gaps (matches `portfolio` `gap-6`) |
| xl | 32px | `8` / `mb-8` | Heading-to-content (matches home `mb-8`), desktop two-column gutter |
| 2xl | 48px | `12` / `py-12` | Page vertical padding (matches home/portfolio `py-12`) |
| 3xl | 64px | `16` | Reserved for large empty-state vertical centering (`py-24` = 96px is the existing empty-state pad in `market-list.tsx` — keep that for full-page empties) |

**Exceptions:**
- Odds bar height stays `h-1.5` (6px) — reuse `odds-display.tsx` verbatim, do not "round to 8".
- Live-status dot is `h-2 w-2` (8px) — on the sm token.
- Touch targets: interactive controls (outcome select, stake stepper, window-toggle buttons,
  primary CTA) must be **≥44px tall** on mobile. The default shadcn `Button size="default"` is
  `h-10` (40px); use `size="lg"` (`h-11` = 44px) for the primary "Place bet" CTA and the
  order-entry controls so the bet flow is thumb-reachable (anticipating Phase 11 ≥360px QA).

## Typography

Exactly 4 sizes, 2 primary weights (matches the existing scale — no new sizes introduced):

| Role | Size | Tailwind | Weight | Line Height | Usage |
|------|------|----------|--------|-------------|-------|
| Display / Page H1 | 30px | `text-3xl` | 600 semibold | tight (`leading-none tracking-tight`, ~1.1) | Market question as page title on detail page (matches `portfolio` H1) |
| Heading / Section H2 | 18px | `text-lg` | 600 semibold (500 medium for sub-sections) | snug (~1.3) | "Resolution criteria", "Price history", "Recent activity", empty/error headings (matches `market-list` `text-lg font-semibold`) |
| Body | 14px | `text-sm` | 400 normal | 1.5 (`leading-normal`/`leading-relaxed` for long criteria text) | Description, resolution-criteria body, activity rows, helper/error copy, money values (matches pervasive `text-sm text-zinc-500`) |
| Label / Micro | 12px | `text-xs` | 600 semibold, `uppercase tracking-wide` for YES/NO; 400 for timestamps | snug (~1.3) | YES/NO labels (matches `odds-display`), "Live"/"Stale" badge text, source badge, relative timestamps |

**Weight rule:** only **400 (normal)** and **600 (semibold)**. `font-medium` (500) is permitted
ONLY where the existing primitives already use it (`button.tsx` `font-medium`, `FormMessage`
`font-medium`, portfolio P&L `font-medium`) — do not introduce 500 on new headings/body.

**Numeric/odds emphasis:** odds percentages render `text-sm font-normal` in the semantic
color (matches `odds-display.tsx` exactly: `text-emerald-700 dark:text-emerald-400` for YES,
`text-rose-700 dark:text-rose-400` for NO). The large "current YES probability" readout on the
detail page uses `text-3xl` (already in the declared scale) for the live price, reusing the same
color tokens — do not add a new accent. (Resolves the UI-checker Dimension-4 FLAG: `text-2xl` is
NOT used; the type scale stays at exactly 4 sizes — 30/18/14/12.)

## Color

The app is a neutral **zinc** system on a **white** dominant surface with exactly two
brand-semantic hues already in use: **emerald** (YES / positive / "live") and **rose/red**
(NO / error / destructive). This phase adds **one** new functional color: **amber** for the
"Stale" connection state. No other new colors.

| Role | Value | Tailwind | Usage |
|------|-------|----------|-------|
| Dominant (60%) | `#ffffff` / `#0a0a0a` dark | `bg-white` / `--background` | Page background, card surfaces (`Card` is `bg-white`) |
| Secondary (30%) | zinc-100 / zinc-900 dark + zinc-200 borders | `bg-zinc-100`, `border-zinc-200` | Card borders, sticky order-panel surface, skeleton fills, muted chips, secondary buttons |
| Text — primary | zinc-950 / zinc-900 | `text-zinc-950` | Headings, question, primary copy |
| Text — muted | zinc-500 / zinc-600 | `text-zinc-500` | Helper text, timestamps, volume/deadline, descriptions |
| Accent (10%) — primary action | zinc-900 / zinc-50 dark | `bg-zinc-900 text-zinc-50` | **Reserved for the primary "Place bet" CTA only** (matches `Button` default + `SourceBadge` House chip). Neutral-by-design — this is a play-money product; accent is restraint, not a bright brand color. |
| Semantic — YES / positive / Live | emerald-500 / emerald-600 / emerald-700 / emerald-400 dark | `bg-emerald-500`, `text-emerald-700 dark:text-emerald-400`, `text-emerald-600` | YES odds + bar (verbatim from `odds-display`), positive P&L (`text-emerald-600` per portfolio), the pulsing **"Live"** dot (`bg-emerald-500`) |
| Semantic — NO / complement | rose-500 / rose-700 / rose-400 dark | `bg-rose-500`, `text-rose-700 dark:text-rose-400` | NO odds + bar (verbatim from `odds-display`). NO is the chart complement and is NOT plotted (binary, YES-only line per CONTEXT Area 2). |
| Semantic — Stale (NEW) | amber-500 / amber-600 / amber-700 / amber-400 dark | `bg-amber-500`, `text-amber-700 dark:text-amber-400`, `border-amber-200` | The ">30s no update" **"Stale"** badge and dot ONLY. Never used elsewhere. |
| Semantic — error / destructive | red-500 / rose-700 / rose-400 dark | `text-red-500`, `text-rose-700` | Inline bet-error copy (matches `FormMessage text-red-500` + `market-list` error `text-rose-700`), `Button variant="destructive"`. |

**Accent reserved for:** the single primary **"Place bet"** CTA and the existing House source
chip. NOT for links, NOT for the chart line, NOT for "all interactive elements". Links use the
existing pattern (underline / `text-zinc-900`), secondary actions use `variant="secondary"` or
`outline`.

**Chart color contract (Recharts):**
- YES probability line: **emerald-600** (`#059669`) stroke, ~2px, with a subtle emerald area
  fill at low opacity (≤15%) acceptable. This keeps the chart on the established YES hue.
- Axes / gridlines: zinc-200 (`#e4e4e7`) light, zinc-700/`#3f3f46` dark, thin/low-contrast.
- Axis tick labels + tooltip text: `text-xs` zinc-500.
- Tooltip surface: `bg-white` + `border-zinc-200` + `shadow-sm` + `rounded-md` (mirror `Card`),
  showing the snapshot timestamp + YES % as a string.
- No red/amber/accent inside the chart. NO line is explicitly out of scope (v1 binary, YES-only).

**Connection-state color mapping (the live indicator state machine):**

| State | Dot | Label color | Label text |
|-------|-----|-------------|------------|
| Connected / fresh | `bg-emerald-500` + slow pulse (`animate-pulse`) | `text-emerald-700 dark:text-emerald-400` | `Live` |
| Stale (>30s, last odds kept visible) | `bg-amber-500` (solid, no pulse) | `text-amber-700 dark:text-amber-400` | `Stale` |
| Reconnecting (socket dropped, backing off) | `bg-amber-500` pulsing | `text-amber-700 dark:text-amber-400` | `Reconnecting…` |

Per the PITFALLS UX rule: **staleness is never silent and odds are never blanked** — the
last-known odds remain rendered behind the Stale/Reconnecting badge.

## Copywriting Contract

All copy is **ENGLISH**, neutral, and play-money-safe. **Hard guardrail (PITFALLS #3, enforced
in existing wallet/portfolio code):** never use "deposit" or casino framing. Use "play money",
"play balance", "stake", "back". Currency renders as the backend string + `PLAY_USD`. Money/odds
are strings on the wire — never `parseFloat` them for display math beyond the existing
percentage rounding.

| Element | Copy |
|---------|------|
| Primary CTA (button) | `Place bet` |
| Primary CTA (after outcome chosen, optional) | `Place bet · YES` / `Place bet · NO` |
| Confirmation modal title | `Confirm your bet` |
| Confirmation modal body (per BET-04 / Phase 5 SC#3) | Three labeled rows: `Stake` → `{stake} PLAY_USD` · `Current odds` → `{yes}% YES / {no}% NO` · `Expected payout` → `{payout} PLAY_USD`. Footer note: `Odds may move before your bet is placed.` |
| Confirmation modal confirm / cancel | Confirm: `Confirm bet` · Cancel: `Cancel` |
| Outcome selector label | `Your prediction` (options: `YES` / `NO`) |
| Stake input label | `Stake (PLAY_USD)` · placeholder `0.00` |
| Resolution-criteria section heading | `Resolution criteria` (always visible — never behind a toggle; the transparency trust signal) |
| Resolution-criteria helper (if locked post-first-bet) | `These criteria are locked and cannot change now that bets have been placed.` |
| Price-history section heading | `Price history` |
| Chart window toggles | `24h` · `7d` · `30d` (default `7d` selected) |
| Recent-activity section heading | `Recent activity` |
| Activity row format (fully anonymized) | `Someone backed {YES|NO} · {amount} PLAY_USD · {relative-time}` (e.g. `Someone backed YES · 50 PLAY_USD · 2m ago`). No username, initials, or id — ever. |
| Live indicator | `Live` / `Stale` / `Reconnecting…` (see state table above) |
| **Empty — market list / home** (reuse existing) | Heading `No markets yet` · Body `Markets will appear here once they are created or synced. Check back soon.` (verbatim from `market-list.tsx`) |
| **Empty — price chart (low data)** | Heading `Not enough price history yet` · Body `Check back soon — the chart appears once this market has a couple of price snapshots.` (renders until ≥2 snapshots) |
| **Empty — recent activity** | Heading `No bets yet` · Body `Be the first to make a prediction on this market.` |
| **Empty — portfolio open** (reuse existing) | `No open positions yet.` |
| **Empty — portfolio settled** (reuse existing) | `No settled positions yet.` |
| **Error — market list** (reuse existing) | Heading `Unable to load markets` · Body `Something went wrong while fetching markets. Try refreshing the page.` (verbatim from `market-list.tsx`) |
| **Error — market detail (404 / not found)** | Heading `Market not found` · Body `This market doesn't exist or is no longer available.` + link `Back to markets`. |
| **Error — market detail (fetch failure)** | Heading `Unable to load this market` · Body `Something went wrong. Try refreshing the page.` |
| **Bet error — insufficient balance** (inline, NOT a toast) | `Not enough play balance. Lower your stake or check your wallet.` |
| **Bet error — market closed** (inline) | `This market is closed and no longer accepting bets.` |
| **Bet error — unverified email (403)** (inline) | `Verify your email to place bets.` + link `Resend verification` → `/verify-email`. |
| **Bet error — banned account** (inline) | `Your account can't place bets right now. Contact support if you think this is a mistake.` |
| **Bet error — below/above stake limits** (inline) | `Stake must be between {min} and {max} PLAY_USD.` |
| **Bet error — generic fallback** (inline, last resort) | `Your bet couldn't be placed. Try again.` — **no generic "transaction failed" toast on the bet flow.** |
| Bet success (inline / non-blocking) | `Bet placed — {stake} PLAY_USD on {YES|NO}.` |

**Destructive actions:** This phase has **no irreversible destructive action**. Placing a bet is
financially committing (locks play money, no cash-out — `sell_position` is 405 by design), so it
is gated by the **confirmation modal** above (stake / current odds / expected payout + explicit
`Confirm bet`). Treat the confirm modal as the destructive-style guard. There is no delete/cancel
of a placed bet in v1.

## Layout & Interaction Contract (phase-specific)

This section is the interaction source of truth for the planner/executor. It encodes the
CONTEXT.md decisions (all 4 grey areas accepted/LOCKED) into a buildable spec.

### Market detail page — `/markets/[slug]` (new route)

- **Page shell:** centered `main`, `max-w-6xl mx-auto px-4 sm:px-6 py-12` (matches home). Market
  question as `text-3xl font-semibold tracking-tight` H1 at top, source badge + Live indicator
  on the same header row.
- **Responsive grid:**
  - **Desktop (`lg:` ≥1024px):** two columns. **Left** (≈2/3, `lg:col-span-2`): question +
    description, **always-visible Resolution criteria**, Price-history chart, Recent-activity
    feed (in that order). **Right** (≈1/3): **sticky** order-entry panel (`lg:sticky lg:top-8`,
    aligns with `py-12`/`mb-8` rhythm).
  - **Mobile/tablet (`<lg`):** single stacked column, order: header → live odds + Live indicator
    → order-entry form → resolution criteria → chart → activity. Must be readable and
    no-horizontal-scroll at **≥360px** (Phase 11 QA target). Use `grid grid-cols-1 lg:grid-cols-3
    gap-8`.
- **Resolution criteria:** rendered in a `Card` (or bordered section), heading `Resolution
  criteria`, body `text-sm leading-relaxed`. **Always visible, never collapsed** — this is the
  transparency trust signal (CONTEXT specifics + PITFALLS).
- **Live odds block:** reuse `OddsDisplay` (YES/NO % + proportional bar) as the canonical odds
  renderer; the Live/Stale indicator sits adjacent (same row or immediately above). On a
  real-time delta, the changed % updates in place with a brief, subtle transition
  (`transition-colors`/opacity, ≤300ms) — no layout shift, no jarring flash.

### Price-history chart (Recharts)

- Default window **7d**; toggles **24h / 7d / 30d** (30d hard cap from Phase 6 snapshots).
- **YES probability line only** (emerald-600), x = snapshot time, y = 0–100% (or 0–1 formatted to
  %). Data and odds arrive as **strings**; format for display, don't do float math beyond
  rendering.
- Server-side downsampling beyond 7d (~hourly buckets) so 30d renders without regression; raw
  5-min snapshots for 24h/7d (per CONTEXT Area 2 — backend concern, but the chart must request a
  `window` param and tolerate variable point density).
- Empty/low-data state (<2 snapshots): the friendly placeholder copy above, occupying the chart's
  area at its normal height (no collapse/jump).
- Responsive: full-width of its column, fixed aspect or min-height (~`h-64`/256px) so the
  skeleton and the chart occupy the same box.

### Order-entry form (built this phase — no existing bet UI)

- Lives in the sticky right panel (desktop) / inline (mobile), inside a `Card`.
- Built with **react-hook-form + zod**, reusing the established pattern in `lib/auth-schemas.ts`
  + `components/ui/form.tsx` (`Form`/`FormField`/`FormItem`/`FormLabel`/`FormControl`/
  `FormMessage`). Add a `BetSchema` (stake: positive decimal-as-string within min/max; outcome:
  `"YES" | "NO"`). Backend is authoritative; client validation is pre-flight only.
- Fields: outcome **Select** (`YES`/`NO`), **Stake** `Input` (`text`/`inputMode="decimal"`,
  label `Stake (PLAY_USD)`), live "Expected payout" preview computed from current odds (display
  only). Primary CTA `Button size="lg"` (44px) labeled `Place bet`.
- **Submit → confirmation modal** (shadcn `Dialog`) showing stake / current odds / expected
  payout + `Confirm bet` / `Cancel` (BET-04 / Phase 5 SC#3). Only on confirm does it POST to the
  existing `backend/app/bets/router.py::place_bet`, cookie-gated player (reuse `lib/auth.ts`
  session-forward pattern; this is a client interaction, so use a Server Action or fetch with the
  session cookie consistent with existing auth flows).
- **Errors are inline + specific** (mapped from backend status → the copy table above), rendered
  via `FormMessage`/an inline alert region with `role="alert"`. **No generic toast** on the bet
  flow.
- Disabled/auth states: if market is `CLOSED` the form is disabled with the closed-market copy;
  if the viewer is unauthenticated, show a `Log in to place a bet` affordance linking to
  `/login` (reuse existing auth routes) rather than a dead form.

### Recent-activity feed

- Last **20** bets on the market (CONTEXT Area 1). **Fully anonymized** — `Someone backed
  {YES|NO} · {amount} PLAY_USD · {relative-time}`. Never expose username/initials/id.
- YES/NO token in each row uses the semantic color (emerald/rose) but stays `text-sm`/`text-xs`;
  rows separated by `·` (xs gap) and a subtle divider or `space-y-2`.
- Empty state: `No bets yet` / `Be the first to make a prediction on this market.`
- v1 is render-on-load + real-time-optional; if a new bet arrives over the socket it MAY prepend,
  but the chart/odds are the primary real-time surface.

### Real-time connection UX (the WebSocket hook)

- One `"use client"` hook (e.g. `hooks/use-market-socket.ts`, new `hooks/` dir) per open detail
  page, subscribed to `/ws/markets/{id|slug}` (public, read-only odds). It owns the connection
  state machine and exponential-backoff reconnect.
- Surfaced UI states drive the **Live / Stale / Reconnecting…** indicator exactly per the color
  + copy tables. Stale fires after **>30s** with no message; odds stay visible.
- The indicator is a small inline cluster: `h-2 w-2` rounded dot + `text-xs` label, placed
  adjacent to the odds block. Use `aria-live="polite"` so the state change is announced to AT
  without stealing focus.

### Loading / skeleton states (Suspense)

- Reuse/extend the existing skeleton vocabulary (`components/ui/skeleton.tsx` +
  `market-list-skeleton.tsx` pattern: zinc skeleton blocks inside the real layout boxes,
  `aria-busy`/`aria-hidden`).
- **Home / market list:** keep existing `MarketListSkeleton` (`Suspense` fallback — already
  wired in `app/page.tsx`).
- **Market detail:** a `MarketDetailSkeleton` mirroring the two-column shell — skeleton for
  title, criteria block, a chart-area block (`h-64`), and the order panel.
- **Portfolio:** add a lightweight skeleton/Suspense boundary consistent with the above (the
  page currently degrades to empty on failure — keep that degradation for errors, add a skeleton
  for the loading window).
- Every loading box must occupy the same dimensions as the resolved content (no layout shift).
