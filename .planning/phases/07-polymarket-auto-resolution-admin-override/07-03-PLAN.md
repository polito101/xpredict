# Plan 07-03: Admin Force-Settle Endpoint (ADM-06)

## Objective
Add a distinct `POST /admin/markets/{market_id}/force-settle` endpoint that force-settles a stuck Polymarket-mirrored market via the unchanged `SettlementService`, capturing the live `umaResolutionStatus` at override time in a `polymarket_admin_override` audit entry.

## Context
This delivers ADM-06: the emergency operator override for mirrored markets that UMA has not (or will not) resolve. Per `07-RESEARCH.md` (Pattern 4) it MUST be a separate endpoint from `/resolve` — `/resolve` is the house-market path (`event_type="settlement.resolved"`), whereas ADM-06 requires a distinct `event_type="polymarket_admin_override"` with extra payload fields. Sharing the endpoint would require conditional logic and produce misleading audit trails. The two-step confirm is a client UX concern (frontend, out of scope here per the RESEARCH open question — Phase 7 ships the API only); the API validates the mandatory justification. It depends only on Plan 07-01 (the migration head) and runs in Wave 2 in parallel with Plan 07-02 (no shared files). The endpoint reuses `SettlementService.resolve_market` unchanged with `actor_user_id=admin.id`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin client → /force-settle API | Privileged money-moving request; must enforce admin auth + input validation |
| Force-settle handler → Gamma API | Reads current UMA status for the audit snapshot (read-only, untrusted) |
| Force-settle handler → ledger (SettlementService) | Money-moving call inside the existing ACID/idempotency contract |

## Security Threats

| Threat ID | Threat | STRIDE | Disposition | Mitigation |
|-----------|--------|--------|-------------|------------|
| T-07-07 | Non-admin invokes force-settle and moves money | Elevation of Privilege | mitigate | `admin: Annotated[User, Depends(current_active_admin)]` enforced (ASVS V4); `test_force_settle_requires_admin` asserts 401 without admin Bearer. |
| T-07-08 | Force-settle a non-Polymarket (house) market, bypassing the house resolve flow | Tampering | mitigate | Validate `market.source == MarketSourceEnum.POLYMARKET.value` before settling; 404 otherwise (RESEARCH Pattern 4). |
| T-07-09 | Override with no/empty justification, breaking the audit trail | Repudiation | mitigate | `ForceSettleRequest.justification: str = Field(min_length=1)` + `extra="forbid"` (ASVS V5); justification persisted in the audit payload. |
| T-07-10 | Audit row lost or settlement corrupted by wrong transaction boundary | Repudiation | mitigate | `await session.rollback()` + capture `admin_id = admin.id` BEFORE the service call (constraint 10 — MissingGreenlet prevention); write the `polymarket_admin_override` row in a SEPARATE `session.begin()` AFTER `resolve_market()` commits (constraints 5, 14 — action-THEN-audit). |
| T-07-SC | Supply-chain: new package introduced | Tampering | accept | No new packages in Phase 7 (RESEARCH Package Legitimacy Audit: none); no install task in this plan. |
</threat_model>
