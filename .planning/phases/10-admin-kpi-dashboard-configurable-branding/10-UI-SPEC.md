> Generated by gsd-ui-researcher in `--auto` mode, verified by gsd-ui-checker.
>
> **Brownfield phase.** The visual language is already shipped (Phases 2/8/9). This contract MIRRORS the
> existing shadcn/ui "new-york" + zinc system — it does **not** introduce a new visual language. Every value

> existing component (cited inline). Consequential assumptions are listed in `## Assumptions` for Pol/planner review.

## Spacing Scale

8-point scale (multiples of 4), matching the shipped components verbatim.

| Token | Value | Tailwind | Usage (grounded in existing code) |
|-------|-------|----------|-----------------------------------|
| xs | 4px | `gap-1` / `space-y-1` | Icon-to-label gaps; window-toggle button group (`flex gap-1`, price-history-chart) |
| sm | 8px | `gap-2` / `space-y-2` | Compact stacks; chart header/body gap |
| md | 16px | `gap-4` / `space-y-4` | Default element + form-field spacing (recharge-form `gap-4`) |
| lg | 24px | `p-6` / `gap-6` | **Card interior padding** (`CardHeader`/`CardContent` are `p-6`); KPI grid gap |
| xl | 32px | `py-8` / `gap-8` | Section breaks between cards-row and chart |
| 2xl | 48px | `py-12` | Page top/bottom padding (admin landing uses `py-12`) |
| 3xl | 64px | `py-16` | Reserved; large empty-state vertical centering |

**Layout container:** `mx-auto max-w-6xl px-6` (the admin shell width, from `app/admin/page.tsx` + `layout.tsx`).

**Exceptions:**
- **44px (`h-11`) minimum touch target** for the DAU window toggle (24h/7d/30d) and any icon-only control —
  mirrors the `h-11` override already used in `price-history-chart.tsx`'s `WindowToggle`. Required for the
  ≥360px reflow / Phase 11 mobile pass.
- **Chart height `h-64` (256px)** fixed parent — mandatory so Recharts `<ResponsiveContainer>` does not collapse
  to 0 (RESEARCH Pitfall 4 / Phase-9 precedent). The 30-day-empty empty state MUST occupy the same `h-64`.

## Typography

Lifted from the shipped scale — `globals.css` (system font), `card.tsx` (`text-2xl`/`text-sm`),
`app/admin/page.tsx` (`text-3xl`), and the `text-sm` body default used across admin components. Exactly
**4 sizes, 2 weights**.

| Role | Size | Weight | Line Height | Tailwind | Where |
|------|------|--------|-------------|----------|-------|
| Display | 30px (`text-3xl`) | 600 semibold | 1.2 (`tracking-tight`, `leading-tight`) | `text-3xl font-semibold tracking-tight` | Page H1 ("Dashboard", "Branding") |
| Heading | 24px (`text-2xl`) | 600 semibold | 1.0–1.2 (`leading-none tracking-tight`) | `text-2xl font-semibold leading-none tracking-tight` | **KPI metric value** + `CardTitle` |
| Body | 16px (`text-base`) | 400 regular | 1.5 | `text-base` / `leading-relaxed` | Paragraph copy, empty-state body |
| Label | 14px (`text-sm`) | 500 medium (form labels) / 400 (muted captions) | 1.5 | `text-sm font-medium` (labels) · `text-sm text-zinc-500` (captions) | Card KPI labels, form labels, helper/caption text |

**Numeric KPI values** use `tabular-nums` (already used in `recharge-form.tsx` for the `$` amount) so figures
align across the card row. Money values render as **strings** (project money-as-string contract) — never
`parseFloat`/`Number()`.

**Weights:** exactly two — **400 (regular)** and **600 (semibold)**. `font-medium` (500) appears in shadcn
form labels and button text as the library default and is the single tolerated intermediate; treat 400/600 as
the design contract and 500 as the inherited shadcn label weight (do not introduce 700/800).

## Color

The system is **neutral zinc on white surfaces** (shadcn new-york default, verbatim from `card.tsx`,
`button.tsx`, `badge.tsx`, `admin/layout.tsx`). Phase 10 adds **two runtime-injected brand tokens** but those
do NOT change the admin chrome — they re-skin the *player* UI.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` white / `zinc-50` (`--background`) | Admin page + card backgrounds (`bg-white`); admin shell body `bg-zinc-50` |
| Secondary (30%) | `zinc-100` / `zinc-200` (borders) / `zinc-900` ink (`--foreground` #171717) | Card borders (`border-zinc-200`), nav bar (`bg-white border-b`), secondary buttons/badges, body text |
| Accent (10%) | `zinc-900` (`#18181b`) — the shadcn default "primary" | Primary CTA button fill (`bg-zinc-900 text-zinc-50`), active nav link, KPI metric ink |
| Positive / chart | `emerald-600` (`#059669`) | Recharts volume series stroke/fill (matches the Phase-9 YES line); positive house-P&L value |
| Destructive | `red-500` (`#ef4444`) | Negative house-P&L value; (no destructive *action* exists this phase — see Copywriting) |
| Muted | `zinc-500` / `zinc-400` | Captions, helper text, placeholders, disabled state |

**Accent reserved for (explicit list — never "all interactive elements"):**
1. The single primary CTA per form (branding form **Save branding** submit button — `Button` default variant, `bg-zinc-900`).
2. The active admin nav link (`font-semibold text-zinc-900 underline underline-offset-4`).
3. KPI metric values (the `text-2xl font-semibold` number ink).

Everything else (secondary buttons, card chrome, badges) stays neutral zinc.

### Runtime brand tokens (player surface — D-10)

Add to `globals.css` (fallback defaults so the player UI is never unstyled if the API is unreachable):

```css
:root { --brand-primary: #4f46e5; --brand-secondary: #0ea5e9; }   /* indigo/sky fallback */
@theme inline {
  --color-brand-primary: var(--brand-primary);     /* usable as bg-brand-primary / text-brand-primary */
  --color-brand-secondary: var(--brand-secondary);
}
```

The player root `layout.tsx` (async Server Component) `await`s `GET /branding/current` and injects
`<style>:root{--brand-primary:<hex>;--brand-secondary:<hex>;}</style>`. **Security contract:** the hex strings
are validated server-side against `^#[0-9a-fA-F]{6}$` BEFORE persist AND BEFORE injection (RESEARCH Pitfall 5 /
Threat: stored CSS injection). The frontend treats the values as opaque validated tokens; it must NOT
re-derive or concatenate untrusted strings into the `<style>` block.

`--brand-primary` is reserved on the player surface for: brand-name accent text, the player header logo
backdrop/accent, and (optionally) the volume-chart stroke so the chart re-skins live. It is NOT a wholesale
replacement of the zinc system — it accents, it does not repaint every surface.

### Accessibility guardrail (operator-chosen palette — design-grounding requirement)

A bad operator palette must never make text unreadable. Contract:

1. **Never** place body text directly on `--brand-primary` without a contrast check. Brand color is used as an
   **accent on neutral surfaces**, or as a **background only when paired with an explicit readable foreground**.
2. Where brand color is a text/icon accent, it sits on white/`zinc-50` — operators pick saturated hexes that
   read fine on white; the `^#[0-9a-fA-F]{6}$` validation guarantees a real 6-digit color (no transparency).
3. **Safe fallback:** if `GET /branding/current` fails or returns no palette, the `:root` defaults
   (`#4f46e5` / `#0ea5e9`) apply — both ≥4.5:1 on white for text use. The player UI is never unbranded-broken.
4. **Header text and primary CTAs on the player surface keep their existing zinc/foreground ink** regardless of
   brand palette — branding accents borders, badges, logo, and chart, not the legibility-critical text layer.
   This is the guardrail that makes "any operator hex" safe without a runtime WCAG calculator in v1.

> Note (assumption A-PALETTE): v1 does **not** compute a runtime WCAG contrast ratio for the chosen palette.
> The guardrail is structural (accent-on-neutral, ink stays zinc, validated 6-digit hex, safe fallback). A
> runtime auto-contrast / accessible-foreground picker is deferred to v2. Flagged for Pol.

### Destructive actions

**None this phase.** No delete / ban / irreversible action exists in Phase 10 (branding is an idempotent
update; KPIs are read-only). Therefore: **no destructive-confirmation dialog is specified.** Saving branding
is a normal (non-destructive) `Button` default-variant submit — it does NOT use the `destructive` variant and
does NOT require a confirm step. (Confirm dialogs in this app are reserved for ban/unban/settlement — out of
scope here.)

> Editorial guardrail (demo-trap branding, ROADMAP pitfall): copy must read like a real SaaS, not a demo.
> Avoid "lorem", "placeholder", "TODO", "coming soon" anywhere on these surfaces — the dashboard and the live
> re-skin are the operator-demo money-shot.

## Responsive Contract

Admin surface is desktop-first, but the **KPI grid MUST reflow to ≥360px** (Phase 11 anticipation, CONTEXT
"responsive (anticipate Phase 11 mobile validation, ≥360px)"):

| Breakpoint | KPI grid | Chart | Toggle/buttons |
|------------|----------|-------|----------------|
| `<640px` (mobile, ≥360px) | `grid-cols-1` (cards stack, no horizontal scroll) | full-width `h-64` | `h-11` (44px) touch targets |
| `sm` (≥640px) | `grid-cols-2` | full-width | — |
| `lg` (≥1024px) | `grid-cols-3` (5 cards → 3+2) | full-width below cards | — |

No horizontal scroll at any width ≥360px. The **player** runtime-theming surfaces (header brand/logo) inherit
the player UI's existing responsive behavior and are bound by PLT-07.
