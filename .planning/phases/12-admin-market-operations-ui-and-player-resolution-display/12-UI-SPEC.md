> Visual and interaction contract for the v1.0 closure phase. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>

### Destructive actions (each + confirmation approach)

| Action | Destructive? | Confirmation |
|--------|-------------|--------------|
| Close market early | Yes (blocks new bets, irreversible to OPEN) | Two-step: button → `Dialog` with consequence copy → `destructive` "Close market". **No** mandatory reason (the API takes none). |
| Resolve market | Yes (settles wallets via ledger) | **Two-step + mandatory justification.** Dialog adds a YES/NO outcome `Select` above the justification `Textarea` (the only structural addition over the ban dialog — `12-RESEARCH.md` Pattern 4). |
| Reverse settlement | Yes (compensating ledger entries) | **Two-step + mandatory justification.** Consequence copy must set the correct expectation (see §Reverse copy guard below). |
| Force-settle | Yes (overrides a stuck Polymarket market) | **Two-step + mandatory justification.** |
| Place / confirm bet (player) | Treated as a commit gate (play money, no cash-out) | Existing `BetConfirmDialog` — unchanged. |

**Reverse copy guard (Pitfall 5, `12-RESEARCH.md`):** the reverse dialog body MUST set the expectation that *reversal restores the pre-settlement state for audit/correction* and must NOT promise "you can resolve again afterward" — re-resolving after a reversal collides on idempotency keys (a known, deferred v1 limitation). Recommended body copy:
> **Reverse this settlement.** This posts compensating ledger entries that return every affected player to their pre-settlement balance and writes an audit entry. It does not re-open the market for a clean re-resolution.

## Open Questions (genuinely-undecided NET-NEW choices — recommended default given)

> These are surfaced for the orchestrator. Each has a recommended default so planning/execution can proceed without blocking.

1. **Information architecture of the admin market-ops page — single detail page vs. list-with-dialogs?**
   - *What's undecided:* whether create/edit/resolve/reverse/force-settle live on a dedicated `/admin/markets/[id]` detail page (RESEARCH's structure) or as dialogs launched from the list rows.
   - *Recommended default:* **`/admin/markets/[id]` detail page** hosts edit + the three settlement action buttons; **create** is a dialog (or `new/page.tsx`) launched from the list header. This mirrors the shipped `/admin/users/[id]` detail-page pattern (where ban/unban/recharge live on the detail page, not the list) and keeps the destructive settlement actions off the high-traffic list. The list stays read+navigate only.

2. **Player resolution-display layout — replace the order panel (right column) vs. a full-width banner above the grid?**
   - *What's undecided:* exact placement of `MarketResolutionPanel`.
   - *Recommended default:* **replace the right-column order panel** (RIGHT `lg:col-span-1`) with the resolution panel, keeping the rest of the detail layout identical. This requires zero grid changes (the slot already exists and is `lg:sticky`), and the "the place where you would have bet now shows how it resolved" mapping reads naturally. A full-width banner would need new layout primitives — avoid.

3. **`resolution_source` storage shape — token-only vs. token + admin display-name snapshot (RESEARCH A2 / Pitfall 6).**
   - *What's undecided (backend, but it drives the panel copy):* the public read has no admin join, so "Operator: {name}" needs the name available at read time.
   - *Recommended default for the UI contract:* the panel reads a stable `resolution_source` token (`"POLYMARKET_UMA"` / `"HOUSE"`) **plus** a resolver display string the backend supplies on `MarketRead` (e.g. `resolution_source` token + the resolving admin's `display_name` snapshot persisted at resolve time). The frontend formats: token `POLYMARKET_UMA` → "Polymarket UMA" (+ source link); token `HOUSE` → "Operator: {display_name}". If the backend ships token-only, the panel falls back to "Operator" without a name. **Planner confirms the backend shape**; the panel copy is written defensively either way.

4. **BET-06 storage (RESEARCH A1) — `markets` columns vs. TenantConfig.**
   - This is a backend/requirement-wording question (RESEARCH recommends columns on `markets`; the requirement text says "via TenantConfig"). It does **not** change the UI: the admin form fields and the player enforcement messaging are identical regardless. Flagged here only so the planner resolves A1 with Pol; the UI contract is storage-agnostic.
