## Architecture decisions (this session + carried)
- **Payout model = "odds locked at placement"** (`.planning/research/ARCHITECTURE.md`): a
  winning bet's gross payout is `stake / price`; loser pays 0; **no house edge in v1** (v2 MAYBE
  per FEATURES.md). `price` is the chosen outcome's price/probability captured at placement.
- **Pure-core-first split**: settlement is split into pure math (`payout.py`, `plan.py` — no I/O,
  unit-tested, reusable UNCHANGED by Phase 7 Polymarket auto-resolution) and a transactional
  shell (`SettlementService`, gated, integration-tested). The pure core takes `price` as an
  INPUT, so it has ZERO Phase 4 coupling.
- **Money quantization**: 4-dp `ROUND_HALF_UP` (`quantize_money`) to match `NUMERIC(18,4)`
  (`app/db/types.py`); Decimals from strings, never float (WAL-05).
- **Idempotent settlement WITHOUT a new constraint**: reuse the existing
  `transfers.idempotency_key UNIQUE` with key `settle:{bet_id}` instead of adding a new
  `(bet_id, event_type)` UNIQUE to `entries` — preserves the single-head invariant during
  parallel dev. (ROADMAP SC#6 mentions the entries constraint as ONE option; this achieves the
  same "zero double-payouts" guarantee on the existing schema.)
- **Additive, low-risk strategy (preserve)**: new modules only, migration-free, validated Phase 3
  ledger reused (`_post_transfer`/`transfer`) and NOT modified.
- **STATE.md / PHASES.md deliberately NOT churned on this branch** to avoid rebase conflicts with
  Pol's parallel Phase 3/4 work on those shared files.
</decisions_made>

(e.g. 0.5 for "50/50", giving 2× payout). Phase 4 exposes `Outcome.current_odds`/`initial_odds`
as `Numeric(8,6)`, and ADM-02 says default "50/50".

odds (≈2.0)? If probability → feed it straight into `compute_payout(stake, outcome.current_odds)`.
If decimal odds → use `price = 1 / current_odds`. Check Phase 4's `MarketService`/HouseAdapter
seed + a created-market row before depending on it.
