Build the ledger SKELETON: the `accounts` / `transfers` / `entries` double-entry schema
(Alembic migration 0003), the ORM models, the wallet constants/exceptions, and a Wave-0
test scaffold proving the DB-level invariants (immutability, `CHECK (balance >= 0)`,
`tenant_id` default, idempotency UNIQUE). This is Wave 1 — everything downstream
(`WalletService`, registration hook, recharge, reads, reconciliation) rides on these tables.

port of the Phase 1 `audit_log` deny-trigger + REVOKE.

Purpose: lock the financial substrate every later money-touching phase inherits; prove the
DB enforces the non-negotiable invariants (WAL-06 append-only double-entry, WAL-08 non-negative
balance) before any service code reads from them.
Output: `app/wallet/{models,constants,exceptions}.py`, migration `0003`, and
`tests/wallet/{conftest,test_models,test_migration_0003}.py`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| application → Postgres | All money truth crosses here; the DB enforces invariants the app relies on |
| migration author → schema | A wrong DDL (missing CHECK, missing trigger) silently weakens every later guarantee |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | accounts.balance going negative | mitigate | `CHECK (balance >= 0)` DB constraint (WAL-08), tested by test_balance_check_rejects_negative |
| T-03-02 | Tampering / Repudiation | UPDATE/DELETE on transfers or entries (ledger tampering) | mitigate | deny-trigger + `REVOKE UPDATE, DELETE` ported from Phase 1 audit_log (WAL-06); tested both ways |
| T-03-03 | Tampering | Money stored/leaked as float (precision drift) | mitigate | `NUMERIC(18,4)` columns + `Mapped[Money]` + money-lint CI gate (PITFALLS #4) |
| T-03-04 | Tampering | Duplicate idempotency_key enabling replay | mitigate | `idempotency_key UNIQUE` constraint; 23505 surfaces (full handling in 03-04) |
| T-03-05 | Elevation / regulatory | A schema FK path that would allow user→user value movement | mitigate | entries.account_id references accounts only; no FK couples two user wallets in one transfer (SC#5/WAL-09 firewall; negative API test in 03-04) |
</threat_model>
